# Agent Update Flow

AI agent がこのプロジェクトを更新するための標準手順。

## 基本原則

- 全データの主キーは `spot.ID`（1〜260）
- `assets/data.js` がマスターデータ
- 変更後は必ず `node --check assets/app.js`
- ローカルテスト `python3 -m http.server 4173` → `curl` で HTTP 200 確認
- **デプロイ前に必ずテストすること**

## 撮影地の追加

### 1. data.js に追加
`assets/data.js` の `spots` 配列にレコード追加。必須フィールド: ID, 地点, 都县, 区域, 类型, 候选等级, 总分100, 交通级别, 预计耗时, スコア内訳7項目。

### 2. 写真取得（優先順）
1. **GANREF** — `https://ganref.jp/spot/photo/jpn/{pref}.html` → LPページサムネイル
2. **Flickr** — 地理座標検索 `lat=X&lon=Y&radius=2` → 位置検証済み
3. **Bing Images** — `murl` パターン（最終手段）
- 保存先: `assets/photos/ganref/{ID}.jpg`
- `photo_meta.js` の `localImageUrl` を更新

### 3. 座標追加
Nominatim で `{日本語名}, {エリア}, {都県}, Japan` 形式で地理编码。`spot_locations.js` を更新。

### 4. 日本語名（必要な場合）
`localizeSpotName()` の aliases に追加。

## 写真の重複チェック
```python
import hashlib
from collections import defaultdict
hash_to_files = defaultdict(list)
for f in Path('assets/photos/ganref').iterdir():
    h = hashlib.md5(f.read_bytes()).hexdigest()
    hash_to_files[h].append(f.name)
# 重複あれば Flickr 地理検索で差し替え
```

## 座標の一括修正
`/tmp/fix_coords_v3.py` を使用。`localizeSpotName()` + `localizeText()` の日本語変換 → Nominatim 検索 → 都県バウンドで検証。

## スコアリング（100点満点）
| 項目 | 満点 | キー |
|---|---|---|
| アクセス | 20 | 交通分20 |
| 画面力 | 25 | 画面强度25 |
| 題材密度 | 15 | 题材密度15 |
| 光/季節 | 15 | 光线季节稳定15 |
| 人流 | 10 | 人流可控10 |
| 機材 | 10 | 器材适配10 |
| リスク | 5 | 风险成本5 |

### Agent 自動採点ルール

新規追加・既存更新のどちらでも、agent は次の順序で自動採点する。主観だけで点を付けず、必ず `source_observations` 相当の根拠を残す。

#### 1. 根拠収集

最低 3 系統の情報を確認する。

1. 写真サイト: GANREF、PHOTOHITO、東京カメラ部、cizucu など  
   作例数、構図の強さ、季節タグ、撮影者の実例を確認。
2. 地図/交通: Google Maps、駅距離、乗換、徒歩、車推奨度  
   東京駅または新宿起点での現実的な移動時間を確認。
3. 公式/観光情報: 都県・市区町村・施設公式  
   開園時間、季節イベント、立入制限、料金、工事、閉鎖リスクを確認。

#### 2. 分項目の採点基準

`交通分20`

- 20: 東京中心部から 0-90 分目安。公共交通で無理なく行ける。
- 15: 90-120 分目安。乗換や徒歩が少し重いが日帰り現実的。
- 9: 120 分前後、車推奨、船・バス本数・山道などの制約が強い。
- 例外: 画面価値が非常に高くても交通点は盛らない。画面価値は `画面强度25` 側で評価する。

`画面强度25`

- 23-25: 一目で主役が強い。富士山、海岸線、鳥居、工場夜景、滝、山岳展望など、単独で作品になりやすい。
- 19-22: 構図を作れば強い。都市夜景、橋、庭園、寺社、鉄道、港湾など。
- 15-18: 日常性が強く、撮影者の腕や条件依存が大きい街歩き・公園・商店街。
- 14 以下: 作例が弱い、主役が曖昧、写真サイトで強い実例が少ない。

`题材密度15`

- 13-15: 1 地点または徒歩圏で複数テーマを組める。例: 海+富士山+鳥居、夜景+建築+街、寺社+庭園+紅葉。
- 10-12: 主題は明確で、周辺にも数カット作れる。
- 7-9: 単一主題寄り。よい条件なら強いが滞在の幅は狭い。
- 6 以下: ほぼ一点撮り。

`光线季节稳定15`

- 13-15: 通年成立。朝夕、夜、雨天、曇天など複数条件で撮れる。
- 10-12: 季節や時間帯の推奨はあるが、外しても成立する。
- 7-9: 桜、紅葉、花、雲海、ダイヤモンド富士など短い旬への依存が強い。
- 6 以下: 条件を外すと写真価値が大きく落ちる。

`人流可控10`

- 8-10: 広い、長焦で抜ける、早朝夜間で避けやすい、三脚や構図変更がしやすい。
- 5-7: 普通。混雑はあるが時間帯や立ち位置で回避可能。
- 3-4: 人気観光地・狭い通路・撮影者集中地点。構図が制限されやすい。
- 2 以下: 混雑、規制、私有地、撮影マナー問題が強い。

`器材适配10`

- 9-10: 24-70、70-200、100-400、広角のいずれかが明確に効く。a7R V/GFX100S で解像感を活かしやすい。
- 7-8: 標準ズーム中心で十分成立。軽装でも撮れる。
- 5-6: 特定レンズや三脚、ND、超望遠などが欲しい。
- 4 以下: 機材制約が強く、手持ち装備との相性が悪い。

`风险成本5`

- 5: 無料または低コスト、開放時間が安定、安全、天候以外の不確実性が低い。
- 4: 軽い料金・営業時間・混雑リスクあり。
- 3: 車、船、潮位、山道、施設休業、イベント依存など注意点あり。
- 1-2: 事前確認必須。閉鎖・通行止め・高コスト・安全面のリスクが強い。

#### 3. 合計とランク

```text
总分100 =
  交通分20
+ 画面强度25
+ 题材密度15
+ 光线季节稳定15
+ 人流可控10
+ 器材适配10
+ 风险成本5
```

- `S`: 86 点以上。わざわざ撮影目的で行く価値が高い。
- `A`: 78-85 点。安定しておすすめできる。
- `B`: 68-77 点。条件が合えば強い。
- `C`: 67 点以下。候補には残すが、目的・季節・天気を選ぶ。

#### 4. 更新時の自動再採点

既存スポットを更新するときは、以下の変化があれば必ず再採点する。

- 新しい写真作例が増えた、または強い作例が見つかった → `画面强度25` / `题材密度15`
- 季節イベント、花期、紅葉、雲海、日の出方向などの情報が増えた → `光线季节稳定15`
- 交通ルート、駅距離、バス本数、車推奨度が変わった → `交通分20`
- 混雑、撮影禁止、工事、閉鎖、三脚禁止、料金変更が見つかった → `人流可控10` / `风险成本5`
- レンズ作例、望遠圧縮、広角構図、三脚要否が明確になった → `器材适配10`

#### 5. 採点ログ

agent は採点後、commit message または更新ログに次を残す。

```text
score_update:
  spot_id:
  name_ja:
  old_total:
  new_total:
  changed_fields:
  evidence_urls:
  note:
```

点数を変更しない場合も、重要な根拠を見つけたら `evidence_urls` に追加する。
